title: Actions
description: The fixed action vocabulary the model picks from each step, the params each takes, index-based vs coordinate clicks, and how failures are reported.

The model never runs arbitrary code. Each step it picks one or more named actions from a fixed menu. Every action is defined in `actions/types.ts` and zod-validated at execution time, so a malformed call fails before it touches the page.

The catalog below is generated from the SDK's action schemas (`@peteqian/browser-agent-sdk`).

## Clicking: index vs coordinates

Most actions target elements by **index**. The page snapshot tags every interactive element with an integer index; the model picks one. This is the preferred path — it survives layout shifts and is what the model sees.

`click`, alone, also supports **coordinates** as a fallback for canvas, WebGL, or other non-DOM surfaces. When using coordinates, both `coordinateX` and `coordinateY` are required — supplying only one fails validation.

<Tabs items={["Index", "Coordinates"]}>

```json tab="Index"
{ "name": "click", "params": { "index": 12 } }
```

```json tab="Coordinates"
{ "name": "click", "params": { "coordinateX": 420, "coordinateY": 260 } }
```

</Tabs>

Beyond raw index/coordinates, the `*_by` actions (`click_by`, `type_by`, `select_by`) and `focus_area` accept a **semantic locator** — `role`, `name`, `text`, `testid`, `label`, `placeholder`, `href`, `dataAttr`, or a cross-snapshot `stableId` — plus an optional `nth`. Use these when you want to re-target the same conceptual element across snapshots.

## How failures are reported

A failed action returns `{ ok: false, message }`. The loop counts it as a failure, increments the consecutive-failure counter, and feeds the message back to the model in the next observation. There are three failure shapes:

1. **Validation error** — params didn't match the zod schema (wrong shape, missing required field, out-of-range value).
2. **CDP error** — element not found, navigation failed, and similar runtime errors.
3. **Timeout** — the action ran longer than `actionTimeoutMs`.

If consecutive failures reach `maxFailures` (default 5), the run terminates with `reason: "max_failures"`. See [Errors and recovery](/docs/guides/errors) for the full reason table.
